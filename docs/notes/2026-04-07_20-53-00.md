A good setup for your case is:

1. CI on every PR (tests, lint, packaging checks).
2. CD on merge to main, deploying directly to your Linux server via your existing self-hosted GitHub runner.
3. Rollback kept available, but not the default action.

For the document conversion workflow itself, treat it like a job system, not a one-shot CLI-only flow.

**Recommended operating model**
1. Intake layer:
  1. New PDF arrives via one of:
    1. Watched input folder on server.
    2. Small internal API endpoint.
    3. Manual enqueue command.
2. Job queue:
  1. Each PDF becomes a job record with job_id, source path, requested options, status, timestamps.
  2. Use Redis-backed queue (or even SQLite queue initially if load is low).
3. Worker process:
  1. Dedicated worker pulls one job at a time (or limited concurrency).
  2. Runs full DocDown pipeline in isolated workdir.
  3. Writes final.md, run.log, summary, and status.
4. Output layer:
  1. Store outputs in a stable artifact path by job_id/date.
  2. Keep original input, final markdown, and logs together for traceability.
5. Status and observability:
  1. Simple dashboard or status command listing queued/running/succeeded/failed jobs.
  2. Metrics to track: queue depth, average runtime, fail rate, largest input size.

**Why this beats ad-hoc runs**
1. Clear separation between deployment lifecycle and conversion workload.
2. Retry and failure isolation become straightforward.
3. Multiple users can submit PDFs without stepping on each other.
4. Easy to add autoscaling/concurrency later.

**Direct-CD but safe**
1. Deploy strategy:
  1. Build artifact on CI.
  2. CD on main pulls artifact and does atomic release switch.
  3. Restart worker service.
2. Keep rollback option:
  1. Keep N previous releases/artifacts.
  2. One command to switch worker back to previous release.
3. Keep schema/config backwards-compatible for one version when possible.

**Conversion workflow policy**
1. Job states:
  1. queued
  2. running
  3. succeeded
  4. failed
  5. retrying
2. Retry policy:
  1. Retry transient failures automatically (for example 2 times).
  2. Do not auto-retry deterministic failures (bad PDF, validation failure).
3. Idempotency:
  1. Same input hash plus same options should map to same output key.
  2. Skip duplicate work when possible.
4. Limits:
  1. Max PDF size/pages.
  2. Per-user or per-source concurrency cap.
  3. Timeout per job/chunk to avoid stuck workers.
